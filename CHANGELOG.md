### Added

- **Per-workspace inference parameters — Phase 4c (MCP server allow-list).** Each workspace can now curate which MCP servers' tools are exposed to its active agent. Last of three set / allow-list axes in Phase 4 of the design doc. Vault schema v10 adds `enabled_mcp_servers TEXT` on `workspaces`, JSON-encoded `[String]?`, with the same three-tier semantics as 4a/4b: `NULL` = inherit; `NULL` on Default = "all servers' tools available"; `[]` = no MCP-derived tools available; explicit list = curated subset. `WorkspaceParamCascade.enabledMCPServers: [String]?` is the ninth axis on the cascade; the resolver itself is a pure pass-through (verified by the new `testEnabledMCPServersResolverIsPurePassthrough` test pinning the architectural decision to keep MCP tool-naming knowledge out of `InferAppCore`). **Composition lives at the consumer.** `ChatViewModel.effectiveEnabledTools` is rewritten to compose Phase 4b's per-tool allow-list with Phase 4c's per-server allow-list using a subtractive rule: when Phase 4c is active, every tool name of the form `mcp.<disallowedServerID>.*` is dropped from the effective set. Tool name parsing mirrors `MCPBuiltinTool.init`'s `mcp.<serverID>.<rawToolName>` construction (`InferAgents/MCP/MCPBuiltinTool.swift:37`); non-MCP tools are unaffected. The rule: both nil → nil (no filter); Phase 4b non-nil → start with that set, else start with the runtime universe (`availableToolNames`); apply Phase 4c subtraction last. **MCP servers run app-globally — this is a per-workspace VISIBILITY filter, not a subprocess gate.** One `MCPHost` instance manages all running servers; the allow-list controls which of their tools surface to the active workspace's agent. New chat-VM helpers: `effectiveEnabledMCPServers: Set<String>?` (cascade resolver), `isMCPServerEnabledInActiveWorkspace(_:)` (drives per-row checkbox state), `setWorkspaceEnabledMCPServers(id:ids:)` (writer; fires `refreshActiveAgentToolSpecs` after the write so the running prompt picks up the new server-derived filter immediately), `toggleMCPServerInAllowList(workspaceId:serverID:universe:)` (same first-toggle-materialises-everything-except-this idiom as 4a/4b). **Eager `availableToolNames` snapshot** added at the end of `bootstrapAgents` (after MCP tools register) so the universe is populated by the first agent activation — closes a timing window where the composer's "Phase 4b nil, Phase 4c non-nil → start with everything" branch could have started with an empty `availableToolNames` if the user hadn't opened the workspace sheet yet. **UI**: `WorkspaceSettingsInline` gains an "Available MCP servers" disclosure mirroring 4a/4b — collapsed by default, header summary, `↺ Default` button, body lists every server in `vm.mcpServers` with per-row checkbox showing `displayName` + monospaced `id` as secondary. 6 new tests in `WorkspaceParamCascadeTests` — fall-through, override, empty-array-is-silence, `hasAnyOverride` extension, **`testEnabledMCPServersIndependentFromEnabledTools`** (catches accidental coupling between the two set-axis fields), and the load-bearing **`testEnabledMCPServersResolverIsPurePassthrough`**. **Phase 4 complete** — all three set-axis allow-lists (agents 4a, tools 4b, MCP servers 4c) shipped; Phase 5 (polish) is the only remaining item in the per-workspace-params arc. Out-of-scope for Phase 4c, Phase 5 candidates: cross-axis UI signalling (an MCP-derived tool's checkbox in "Available tools" should grey out when its server is allow-listed out, with a tooltip explaining); registry-side runtime gate (defense-in-depth — if the model hallucinates a disabled tool name, `toolRegistry.invoke` would still execute it); bulk operations (enable-all / disable-all / import-from-another-workspace).

- **Per-workspace inference parameters — Phase 4b (tool allow-list).** Each workspace can now curate which tools are exposed to its active agent. Replication of Phase 4a's JSON-set-cascade pattern with one deliberate semantic divergence: **no safety net**. Where Phase 4a's `effectiveEnabledAgents` always insists on `DefaultAgent.id`, an explicit `[]` for tools genuinely means "no tools available" — that's a legitimate workspace shape (private / security-sensitive contexts where the user explicitly doesn't want web fetches, fs reads, etc. happening). Vault schema v9 adds `enabled_tools TEXT` on `workspaces`, JSON-encoded `[String]?`, with the same three-tier semantics: `NULL` = inherit; `NULL` on Default = "all tools available"; `[]` = workspace-silenced; explicit list = curated subset. `WorkspaceParamCascade.enabledTools: [String]?` is the eighth axis on the cascade; `Vault.setWorkspaceParams.enabledTools` is the eighth `ParamWrite` parameter; `WorkspaceSummary.enabledTools` propagated through the row-mapper using the existing `decodeStringArray` helper. **AgentController API extended.** `switchAgent(...)`, `activateForSegment(...)`, and the private `activate(...)` all take a new `enabledTools: Set<String>? = nil` parameter; the agent's declared `toolsAvailable(for: ctx)` result is intersected with the allow-list (by `ToolSpec.name`) before being assigned to `activeToolSpecs`. **Filter happens at spec-assignment time, not at registry-invoke time** — gating the prompt is more effective than gating the runtime call: the model never sees disabled tools and can't hallucinate calls to them. The default-nil signature keeps every existing `AgentController` test green without modification. `activateForSegment` also gains a `forceRefresh: Bool = false` parameter that bypasses its same-id no-op guard, used by the new `ChatViewModel.refreshActiveAgentToolSpecs()` to re-compute `activeToolSpecs` against the current workspace allow-list **without** changing the active agent — fires after every `setWorkspaceEnabledTools` write so the running prompt picks up the filter immediately rather than waiting for the next agent switch. **Chat-VM helpers.** `effectiveEnabledTools: Set<String>?` (resolves the cascade), `isToolEnabledInActiveWorkspace(_:)` (drives the per-row checkbox state), `setWorkspaceEnabledTools(id:ids:)` (writer + refresh + force-recompose), `toggleToolInAllowList(workspaceId:toolName:universe:)` (first-toggle materialises "everything except this" rather than "only this," matching Phase 4a's idiom). New `availableToolNames: [String]` `@Observable` snapshot on the chat-VM populated by `refreshAvailableToolNames()` (awaits `toolRegistry.allSpecs()` — actor hop) so the workspace sheet's checkbox list reflects the latest tool catalogue including MCP-discovered tools that register at runtime. **Existing activation call sites threaded** — `ChatViewModel.switchAgent`, `Generation.swift`'s segment-activation, and Phase 3's `recomposeActiveAgentFromActiveWorkspace` all now pass `effectiveEnabledTools` so the filter applies on every activation path, not just the explicit refresh. **UI**: `WorkspaceSettingsInline` gains an "Available tools" disclosure mirroring Phase 4a's "Available agents" — same collapsed-by-default + header summary + `↺ Default` button shape; body lists every tool name in `availableToolNames` with a per-row checkbox in monospaced caption font (tool names are identifiers like `http.fetch`, `fs.read` — readability matters). Snapshot refresh hooked into `seedFields`'s `force` path so MCP tools registered at runtime appear in the editor on next sheet-open. 6 new tests in `WorkspaceParamCascadeTests` cover fall-through, override, **empty-array-really-means-no-tools** (the no-safety-net contract pinned), both-nil-produces-nil, **enabled-tools-independent-from-enabled-agents** (catches refactors that accidentally couple the two set-axis fields), `hasAnyOverride`-includes-enabledTools. Out-of-scope by design: registry-side runtime gate (defense-in-depth Phase 5 polish — if the model hallucinates a disabled tool name, `toolRegistry.invoke` would still happily run it; the prompt-side filter is the user-visible barrier, the runtime gate is hardening). Phase 4c (`enabled_mcp_servers`) is the last set-axis; gating happens at MCP-host bootstrap rather than agent activation, so it's a different wire-up than 4a/4b but the JSON-set-cascade data layer pattern carries forward unchanged.

- **Per-workspace inference parameters — Phase 4a (agent allow-list).** Each workspace can now curate which agents are visible in its picker. First of three set / allow-list axes in Phase 4 of the design doc; tools (`enabled_tools`) and MCP servers (`enabled_mcp_servers`) are sequenced behind this and reuse the same JSON-set-cascade pattern. Vault schema v8 adds `enabled_agents TEXT` on `workspaces`, JSON-encoded `[String]?`. Three semantic levels: `NULL` = inherit from cascade; `NULL` on Default = "every agent in the registry available" (the implicit "everything" floor for set axes); explicit `[]` = workspace-silenced state; explicit `'["coder","researcher"]'` = curated subset. `Vault.setWorkspaceParams` JSON-encodes on write; the row-mapper's new `decodeStringArray` helper parses defensively (malformed JSON → nil → "no allow-list" rather than crashing the row read; the column is user-data-shaped and could be hand-edited via a sqlite browser). `WorkspaceParamCascade` (in `InferAppCore`) gains `enabledAgents: [String]?` as a seventh axis — same `??` resolver semantics work because the unit of override is the whole array, not per-id signals. **DefaultAgent-always-allowed safety net** is enforced at the chat-VM consumer (`effectiveEnabledAgents: Set<AgentID>?`), NOT in the cascade resolver — keeping `DefaultAgent` knowledge out of `InferAppCore` preserves the dependency graph (the one test pinning this contract is `testEnabledAgentsResolverDoesNotInjectSafetyNet`). The safety net guarantees the user can never lock themselves out of a workspace; even an explicit `[]` row resolves to `{DefaultAgent.id}` at the consumer. New chat-VM helpers: `effectiveEnabledAgents`, `isAgentEnabledInActiveWorkspace(_:)`, `isVisibleAgent(_:)` (combines the existing backend-compatibility check with the new allow-list check — replaces bare `isCompatible` filters in `AgentPickerMenu` and `InferApp.swift`'s keyboard-shortcut menu), `setWorkspaceEnabledAgents(id:ids:)` (direct writer — nil clears, list pins, refresh + Phase 3 recompose fire after each write so a now-disallowed active agent gracefully degrades to Default), `toggleAgentInAllowList(workspaceId:agentId:)` (first-toggle materialises "everything except this" rather than "only this," matching the user's mental model when starting from the implicit "all available"; subsequent toggles flip individual ids). **Phase 3 graceful-degradation extended** — `recomposeActiveAgentFromActiveWorkspace`'s gate now also checks `isAgentEnabledInActiveWorkspace`, so an active agent that gets allow-listed-out automatically falls back to Default with the same warning-log diagnostic Phase 3 emits for backend-incompatible / registry-evicted cases. The third trigger condition I'd documented in Phase 3's docstring (allow-list excluding required tools) is now live. **UI**: `WorkspaceSettingsInline` gains an "Available agents" disclosure below "Output directory" — collapsed by default, header summarises the state with `(N allowed)` / `(inheriting Default)` / `(everything available)` and a `↺ Default` button when this row's `enabled_agents` is non-NULL. Unfolded body lists every known agent with a per-row checkbox (`Toggle.checkbox`); `DefaultAgent` is rendered as `force-on` and disabled-from-toggling with an explanatory tooltip ("DefaultAgent is always available — the safety net so you can never lock yourself out of a workspace"). Right-sidebar `WorkspaceOverrideClearButton` (Phase 1) NOT extended for the allow-list — different UI shape (per-row checkbox grid vs slider) and different surface (sheet vs sidebar) made reuse over-engineered. 7 new tests in `WorkspaceParamCascadeTests` cover fall-through, override, **empty-array-as-explicit-silence (the set-axis analogue of empty-string-is-an-override)**, both-nil-produces-nil (distinct from the empty-array case — "no allow-list anywhere" vs "explicitly silenced"), independent-axis composition with the other six fields, `hasAnyOverride`-includes-enabledAgents (incl. empty-array case), and the load-bearing `testEnabledAgentsResolverDoesNotInjectSafetyNet` that pins the architectural decision to keep `DefaultAgent` knowledge out of `InferAppCore`. Phase 4b (`enabled_tools`) and 4c (`enabled_mcp_servers`) still sequenced behind this — both pure replication of the JSON-set-cascade pattern just landed (no equivalent of the DefaultAgent safety net since there's nothing analogous in tools / MCP).

- **Per-workspace inference parameters — Phase 3 (active agent / persona).** The user's active agent selection now persists per-workspace and survives app restart. Vault schema v7 adds `active_agent_id TEXT` nullable column on `workspaces`; pure additive ALTER, no migration data write. Default's row stays `NULL` on first launch — there's no legacy `UserDefaults` source to seed from since `activeAgentId` was never persisted before this migration (it defaulted to `DefaultAgent.id` on every cold launch). `WorkspaceParamCascade` (in `InferAppCore`) extended with the new `activeAgentId: String?` axis as a sixth independent field; `resolve(...)` honours it; `hasAnyOverride` includes it. Stored as the raw `AgentID.rawValue` (a String) so the InferAppCore type stays free of the `InferAgents` dep. `Vault.setWorkspaceParams` gains a sixth `activeAgentId: ParamWrite<String?>` parameter; `WorkspaceSummary.activeAgentId` propagated through the row-mapper. **Recompose flow.** New `ChatViewModel.recomposeActiveAgentFromActiveWorkspace(insertDivider:)` resolves the effective active-agent id from the cascade (active workspace's column → Default's column → no-op leaving the controller at its boot default of `DefaultAgent.id`), finds the matching listing in `availableAgents`, and dispatches to one of two activation paths on the existing `AgentController`: `switchAgent(...)` for workspace switches (full path with transcript divider + vault conversation invalidation, identical to a manual user-driven switch) or `activateForSegment(...)` for boot (silent runner-state push only — no transcript effect since there's no conversation to divide yet). `switchWorkspace(to:)` calls it with `insertDivider: true` after `recomposeSettingsFromActiveWorkspace`; `refreshWorkspaces`'s post-load `MainActor.run` block calls it with `insertDivider: false` so the user's last-saved agent restores quietly at boot. **Graceful degradation.** If the resolved id isn't in `availableAgents` (persona was deleted out-of-band) or isn't compatible with the current backend (vision-only agent saved while llama is loaded, or — once Phase 4 lands — required tools no longer in the workspace's allow-list), the recompose falls back to `DefaultAgent.id` rather than ending up in a nil-agent state, with a `warning`-level log line naming the unreachable id and the active backend so the cause is diagnosable from console output. **Persistence on user-driven switch.** `ChatViewModel.switchAgent(to: listing)` now also writes `listing.id.rawValue` to the active workspace's `active_agent_id` column after the controller's effects apply — gated on `effects.isEmpty == false` so we don't write a no-op row when the controller refused the switch (incompatible / already-active). 5 new tests in `WorkspaceParamCascadeTests` cover fall-through, active-overrides-default, independent-axis composition with the other five cascade fields, empty-string-is-override semantics (the chat-VM's recompose guards `rawId.isEmpty` separately), and `hasAnyOverride`-includes-activeAgentId. **Deliberately no new UI affordance** — the existing agent picker (header dropdown / sidebar Agents tab) writes per-workspace automatically; a `↺ Default` button doesn't fit since the picker already lets the user choose any agent including Default. **Latent ordering hazard worth noting**: `availableAgents` is populated by `bootstrapAgents()` async; if `recomposeActiveAgentFromActiveWorkspace(insertDivider: false)` fires from `refreshWorkspaces`'s post-load block before `bootstrapAgents` finishes loading the registry, the recompose finds an empty `availableAgents` and falls through to `DefaultAgent.id` — silently overwriting a saved selection. Chat-VM init order today is `bootstrapAgents()` first, then `refreshWorkspaces()`, but neither awaits the other, so the race is real. Acceptable for Phase 3 because the worst case is a one-launch wrong-agent default that the user can re-pick (which then re-persists); a follow-up audit can sequence the two if user-saved selections start "sticking" inconsistently across launches.

- **Per-workspace inference parameters — Phase 2 (output directory).** Generated artifacts (Stable Diffusion images today; transcript exports later) now write to a per-workspace path. Vault schema v6 adds a single `output_directory TEXT` nullable column on `workspaces`; pure additive ALTER, no migration data write. Default's row stays `NULL` on first launch — the chat-VM substitutes the legacy hardcoded `Application Support/Infer/Generated Images/` path at read time so existing users see no change. `WorkspaceParamCascade` (in `InferAppCore`) extended with the new `outputDirectory: String?` axis; `resolve(...)` honours it independently of the four sampling fields; `hasAnyOverride` includes it. `Vault.setWorkspaceParams` gains a fifth `outputDirectory: ParamWrite<String?>` parameter; `WorkspaceSummary.outputDirectory` propagated through the row-mapper. New `ChatViewModel.effectiveOutputDirectory: URL` resolves the active-workspace → Default → legacy chain, with tilde-expansion at use time so a stored `~/Pictures/Infer/` resolves against the current `$HOME`. New `ChatViewModel.legacyOutputDirectory()` static helper extracts the hardcoded final fallback as one source of truth shared by the runtime and the UI placeholder. New `ChatViewModel.setWorkspaceOutputDirectory(id:path:)` is a dedicated writer (parallel to `setWorkspaceDataFolder`'s pattern, deliberately separate from `applySettings` since `outputDirectory` isn't on `InferSettings` — it's a workspace-identity setting, not a runtime-tuned slider, and bloating `InferSettings` for the four other consumers that don't care wasn't worth it). `sdOutputDirectory` (`StableDiffusion.swift:20`) is now a one-liner that delegates to `effectiveOutputDirectory`; behaviour-preserving for users who haven't set an override. **UI**: `WorkspaceSettingsInline` gains an "Output directory" field below "Data folder" — `TextField` with the `effectiveOutputDirectoryPlaceholder` showing what the next generation would actually write (`Default: <path>` when Default has one set, else the legacy path), a folder-picker button, an Apply-on-change link, and a `↺ Default` button inline next to the label that appears only when this workspace's `output_directory` column is non-NULL. Clearing writes nil, which triggers the cascade fall-through. Right-sidebar `WorkspaceOverrideClearButton` (Phase 1) intentionally NOT extended for outputDirectory — different UI shape (TextField vs slider) and different surface (sheet vs sidebar) made one shared component over-engineered. 5 new tests in `WorkspaceParamCascadeTests` cover outputDirectory fall-through, active override, partial-cascade independence (outputDirectory acts as an independent axis from the four Phase 1 fields), empty-string-is-an-override semantics (the `setWorkspaceOutputDirectory` writer trims empty-to-nil before persistence so this case shouldn't reach the store, but the cascade resolver itself honours what it's given), and `hasAnyOverride`-includes-outputDirectory. Phases 3-5 (persona, MCP/tool/agent allow-lists, polish) still sequenced behind this in the design doc.

- **Per-workspace inference parameters — Phase 1 (sampling + system prompt).** New design doc at `docs/dev/per-workspace-params.md` scopes the full "workspace as runtime context" feature across five phases (sampling params, output dir + wiki budget, persona, MCP/tool/agent allow-lists, polish); this PR ships Phase 1 only — `systemPrompt`, `temperature`, `topP`, `maxTokens` go per-workspace via four nullable columns on the `workspaces` table (vault schema v5 migration: pure additive ALTER, no data rewrite). Override semantics are sparse: `NULL` means "inherit from the Default workspace's row," and **the Default workspace's row IS the global floor** in the new model — no separate `UserDefaults`-backed global slot. The legacy `infer.systemPrompt` / `.temperature` / `.topP` / `.maxTokens` keys exist only as the migration source on first v5 launch (idempotent: the chat-VM populates Default's columns from those keys if they're all NULL after the migration runs, then never reads them again). New `WorkspaceParamCascade` value type in `InferAppCore` (`Sources/InferAppCore/WorkspaceParamCascade.swift`) holds the two-layer cascade resolver as a pure `Sendable` function — chat-VM's `composeEffectiveSettings` adapts `WorkspaceSummary` rows into cascade values and lays the result over an `InferSettings` populated from `UserDefaults` (which still owns ~60 other settings fields not yet per-workspace). New `Vault.setWorkspaceParams(id:systemPrompt:temperature:topP:maxTokens:)` writer with a `ParamWrite<T>` three-state enum (`.unchanged` / `.value(T)`) so callers can update any subset of fields in one query without confusing "skip this field" with "set to NULL." `applySettings(new)` diffs against `previous` and only persists per-workspace fields that actually changed — preserves the "inherit live from Default" promise (a workspace's row stays NULL where the user didn't touch it). Workspace switch fires `recomposeSettingsFromActiveWorkspace(applyToRunners: true)` which routes through the same `applySettings` runner-update path a slider drag takes, so the next turn runs against the new effective values. **Right sidebar UI**: each of Temperature, Top P, Max tokens, and the System prompt disclosure gains a per-field `↺ Default` button (`WorkspaceOverrideClearButton` in `Sidebar/SidebarComponents.swift`) that renders only when (a) a non-Default workspace is active and (b) that workspace's column for the field is non-NULL. Click clears the override; the sidebar's new `.onChange(of: vm.settings)` syncs the local draft to the new effective values so the slider snaps back to Default's value. New chat-VM helpers: `clearWorkspaceParamOverride(_ field: WorkspaceParamField)` (fires `setWorkspaceParams(.value(nil))` for the named column), `activeWorkspaceOverrides(_:)` (drives the button's render gate), `WorkspaceParamField` enum (`.systemPrompt` / `.temperature` / `.topP` / `.maxTokens`). 11 new tests in `Tests/InferAppCoreTests/WorkspaceParamCascadeTests.swift` cover both-nil → all-nil, fall-through to defaults, active wins, per-field override, empty-string + zero as legitimate overrides (NOT nil-equivalent), idempotent same-row collapse, partial defaults, `hasAnyOverride` semantics. Vault round-trip itself isn't unit-tested — vault lives in the executable target and can't be `@testable`-imported without restructuring; the cascade resolver tests cover the substantive logic, vault behaviour is verified via running the app. Phases 2-5 (output directory, wiki context budget, persona, MCP/tool/agent allow-lists) sequenced behind Phase 1 in the design doc; each phase reuses Phase 1's pattern without rework.

- **`InferAppCore` library target — testable nucleus for the chat-VM's runner-driving slice.** New target at `projects/infer/Sources/InferAppCore/` (no MLX / llama / SwiftUI deps, runs under plain `swift test` without the Metal toolchain). Three types: **(1)** `ChatRunner` protocol — narrow chat-side surface (`setHistory`, `respondToUser`, `requestStop`, `resetConversation`, `rewindLastTurn`) distilled from the actual `LlamaRunner` / `MLXRunner` / `CloudRunner` chat path (`Generation.swift`, `Transcript.swift`); system prompt accepted in-band as a leading `.system` `ChatTurn` so each adapter routes it to the right slot (Llama's dedicated `setSystemPrompt`, MLX/Cloud's in-band history). Sampling stays out-of-band per the chat-VM's existing `applySettings` model — only `maxTokens` flows through per call. **(2)** `TranscriptStore` — value-typed `[TranscriptEntry]` with `appendUser`, `beginAssistant`, `appendChunk`, `setText`, `editAndResend`, `regenerate`, `reset`, `turnsForHistory`. `Sendable`, no SwiftUI dependency. **(3)** `ChatTurnDriver.runOneTurn(runner:priorHistory:userText:maxTokens:onChunk:)` — async kernel encoding the F-3 ordering invariant explicitly (`setHistory` strictly before `respondToUser`); forwards each streamed chunk to an optional callback, returns the assembled reply on completion. New `InferAppCoreTests` target with `MockChatRunner` (scriptable actor recording every call in order — analogue of the existing `MockAgentRunner` for the chat-side surface) and **34 tests** covering store semantics (16: append flow, edit-and-resend with image preservation + late-chunk no-op, regenerate, history snapshot), mock streaming + cancellation + error injection (6), cross-type integration with the real F-3 / F-8 ordering scenarios (4), and `ChatTurnDriver` contract (8). Sub-3-second `make test` unchanged.

### Changed

- **Workspace reset is now narrowly scoped to inference parameters; wiki and RAG corpus are preserved.** `ChatViewModel.resetWorkspace(id:)` (`Workspaces.swift:307`) previously deleted the workspace's vector-store entries and removed its on-disk wiki directory, treating reset as a content wipe. Revised after Phase 1 design review: wiki content is curated user authoring, not generated state, and conflating it with param-reset is surprising. New body clears the four per-workspace param columns to NULL via `setWorkspaceParams` and re-runs `recomposeSettingsFromActiveWorkspace(applyToRunners: true)` so the runner picks up the post-clear effective values; no filesystem touches, no RAG mutation, no wiki mutation. For Default workspace this restores the hard-coded `InferSettings.defaults` floor (since Default IS the globals layer and there's no further fallback). For non-Default it clears the sparse overrides and the workspace falls back to inheriting from Default — the bulk version of the per-field `↺ Default` buttons in the sidebar. UI updated to match: `WorkspaceSettingsInline` button label is now "Reset parameters" (was "Reset workspace"), the destructive role + orange tint dropped from the button itself (the operation is no longer destructive of content; the alert dialog still uses `.destructive` on the confirm action because it does clobber the user's tuning), alert title now "Reset parameters to defaults?", message: "Clears any per-workspace overrides for system prompt, temperature, top-p, and max tokens. Wiki pages, RAG corpus, conversations, and the data folder setting stay." Today this UI only renders for the Default workspace (non-Default shows Delete instead); a non-Default bulk-reset button is a one-line addition for later if the per-field buttons prove insufficient.

- **`ChatViewModel` drives every per-backend chat-side switch through `activeChatRunner: any ChatRunner`.** Three concrete runner adapter files (`LlamaRunner+ChatRunner.swift`, `MLXRunner+ChatRunner.swift`, `CloudRunner+ChatRunner.swift`) make the existing `LlamaRunner` / `MLXRunner` / `CloudRunner` actors conform to `InferAppCore.ChatRunner` — Llama's adapter peels system turns off the front of `setHistory(turns)` and routes them to its dedicated `setSystemPrompt`; MLX's preserves per-turn `imageURLs` for VLM models; Cloud's drops system / tool turns at the boundary (Cloud's underlying `setHistory` only accepts user / assistant — system prompt managed via `configure` / `updateSettings`). New `activeChatRunner` computed property on `ChatViewModel` returns the right adapter for the current `Backend`. Six backend switches collapsed across `ChatViewModel.reset()`, `Transcript.compactKVForVisibleHistory`, `Transcript.restoreBackendHistory`, `Generation.requestStopCurrentRunner`, `Generation.stop`, `Generation.regenerateLast`, and `Generation.editLastUserMessage` — each goes from a 4-14 line `switch backend` block to one `await runner.X()` call. `restoreBackendHistory`'s Llama-specific throw-fallback-to-reset semantics generalize cleanly (only Llama can throw; the catch is a no-op for MLX/Cloud). New `Transcript.chatTurns(from:)` helper builds the `[ChatTurn]` snapshot uniformly, filtering system / `agentDivider` rows and preserving per-turn image attachments so MLX VLM history survives regenerate / load-transcript / KV compaction without a per-backend tuple shape. `send()` deliberately NOT migrated — verified that all three runners accumulate state across successive `respondToUser` calls (Llama keeps its KV cache, MLX rebuilds session from internal history, Cloud appends to in-memory messages); calling `setHistory` every turn would force multi-hundred-ms-per-turn KV re-prefill on local models. Documented at the top of `send()` with a pointer to `ChatTurnDriver` as the kernel for the *other* (rebuild-then-send) shape. The `sendUserMessage` switch in the live decode loop also stays per-backend because the MLX branch carries `imageURLs:` for VLM models — the protocol doesn't yet lift that parameter.

- **`TranscriptStore` mirror on `ChatViewModel`, rebuilt at well-defined sync points.** New `@ObservationIgnored var transcriptStore = TranscriptStore()` alongside the SwiftUI-bound `messages: [ChatMessage]`, plus `Transcript.makeTranscriptEntry(_:)` (UI → value mapping, lossy on agent attribution / step traces / RAG provenance / `<think>` disclosure state) and `Transcript.syncTranscriptMirror()` which rebuilds the store from `messages` filtering `agentDivider` rows. Sync points wired in this PR: `reset()`, `loadTranscript()`, `send()` (after user + assistant placeholder appends), `unspoolLastTurn()` (after the trailing user+assistant pair pops — covers regenerate and edit-and-resend). End-of-turn-accurate, not per-token-accurate; the streaming chunk-text writes inside `Generation.swift` (~lines 495, 526, 588, 902, 940) are intentionally not mirrored per-chunk to avoid touching deep streaming code without a concrete bug to chase. The mirror unlocks future flows that consume `turnsForHistory()` directly instead of building tuple snapshots inline.

- **Per-step shutdown timeout with stderr logging — eliminates the relaunch-crash class.** `AppDelegate.applicationWillTerminate` (`InferApp.swift:163`) rewritten from a single 2-second `DispatchSemaphore` wrapping all 17 sequential async cleanup operations to a per-step loop: each step (`llama.requestStop` → `llama.shutdown` → `mlx.requestStop` → ... → `vault.shutdown`) runs on its own detached task with a 300ms budget; timeouts log to stderr (`[shutdown] 'mcpHost.shutdown' timed out after 300ms — proceeding (resource may leak)`) and slow completions (≥100ms) get a less-loud line (`[shutdown] 'vault.shutdown' took 142ms`). `llama_backend_free()` always runs at the end regardless of whether earlier llama steps timed out — the prior implementation's failure mode was a stuck step (MCP subprocess wedged on stdio, HF download cancel, SQLite WAL checkpoint) starving the rest of the chain, leaving the global llama backend in a stale state and `EXC_BAD_ACCESS`-ing on the next launch's `llama_backend_init`. Step order preserved: each runner's `requestStop` precedes its `shutdown`; vault shutdown last so prior steps' WAL writes get checkpointed. Worst-case wall time is bounded by `17 × 300ms = 5.1s` (vs. 2s before); realistic median is <100ms because every step typically completes in tens of ms — only stuck steps consume their full budget. ### Fixed

- **Reasoning models leaked thinking into the visible reply when the model wrote `</think>` as literal prose.** `ThinkBlockStreamFilter` was purely string-matching `<think>` / `</think>` substrings, so when a Qwen-3 / DeepSeek-R1 model emitted something like `the initial message is just "</think>", which doesn't make sense` *inside* its reasoning, the filter saw the literal close tag, flipped to "not thinking", and routed everything that followed (still part of the model's actual reasoning) into the visible reply body. Replaced with token-ID-authoritative detection. `LlamaRunner` probes for `<think>` / `</think>` as single special tokens at load time via `detectSpecialToken(vocab:text:)` — calls `tokenize(...)` with `parse_special=true` and accepts the result only when it tokenises to exactly one token (reasoning models yield 1, base models yield multiple → nil). When those token IDs fire during decode, `runDecodeLoop` injects PUA sentinel chars (`U+E600` for open, `U+E601` for close — Unicode guarantees no defined meaning in the Private-Use Area, so a model emitting these by chance is a non-issue) into the stream instead of the special token's decoded bytes. `ThinkBlockStreamFilter` flips into authoritative "sentinel mode" on the first sentinel and from that point treats surface-form `<think>` / `</think>` strings as passthrough text — only the runner's signal toggles `inThink`. Backends that don't emit sentinels (cloud, MLX, base llama models without special think tokens) keep the legacy string-match path automatically; the filter only swaps modes once it actually sees a sentinel. Four new sentinel-mode tests cover open/close happy path, the screenshot-repro literal-`</think>`-mid-thinking case, sentinel-mode persistence across feeds, and mid-piece sentinel handling.

- **Llama streaming corrupted multi-byte UTF-8 sequences (emojis rendered as `��`).** llama.cpp's BPE tokenizer can split a 4-byte UTF-8 emoji across two tokens (`👋` = `F0 9F 91 8B` → `[F0 9F]` + `[91 8B]`). The old `piece(vocab:token:)` did `String(decoding: bytes, as: UTF8.self)` per token, and `String(decoding:)` *replaces* invalid sequences with U+FFFD instead of failing — so each fragment came back as one or two `�` chars. Replaced with a byte-buffered emit at the `runDecodeLoop` level: `pieceBytes(vocab:token:)` returns raw `[UInt8]`, and a new `safeUTF8Boundary(_:)` walks back ≤4 bytes (the longest UTF-8 sequence) looking for the most recent lead, checks whether its expected continuation count is satisfied, and returns the cut point. `pending` carries the trailing partial sequence into the next iteration. Both the streaming chat path and `generateOneShot` (used by HyDE in `RAGPipeline.swift`) funnel through `runDecodeLoop`, so one fix covers both. Pathological input (4 continuation bytes with no preceding lead — never produced by a well-formed tokenizer) flushes as-is rather than accumulating forever.

Removing an entry strips the plugin from the binary entirely (no compiled bytes); `enabled: false` keeps it compiled-in but skips registration (lands in a follow-up PR — currently equivalent to removal). Per-developer `projects/plugins/plugins.local.json` shadow-merges over the tracked file (gitignored) for opting out of heavy plugins locally without touching tracked state. The leaf-API package shape was the load-bearing architectural choice: SPM rejects package-level cycles, and a flat "Infer ↔ plugin" dep is a cycle (Infer needs plugin types to compile; plugin needs `BuiltinTool` etc. to compile). The leaf package breaks the cycle and means each plugin compiles to its own static library — touching `Infer.swift` doesn't recompile any plugin, and touching a plugin doesn't recompile `InferAgents`. Naming convention pinned: `projects/plugins/plugin_<snake>/` directory + module + `import`, `<UpperCamel>Plugin` Swift type inside, `<snake>` (no `plugin_` prefix) as the `id` in `plugins.json` and UI labels — three names, three jobs (filesystem identifier, Swift code, user/config identifier). No fallback to system `python3` — that defeats the point of shipping a curated runtime with the app's required packages baked in. A missing framework throws `PythonToolsError.frameworkNotFound` with a "run `make fetch-python`" hint, which `PluginLoader` catches → the rest of the host launches normally and `python.*` tools are absent from the registry rather than crashing at first invoke. **Subprocess model, not in-process libpython.** `Foundation.Process` spawns the framework's `python3 -` reading code from stdin; `Pipe`s capture stdout/stderr separately; per-invocation temp working dir, removed on return (no persistence between calls — tools that need persistence write to `~/Documents` via `fs.write`). Output is hand-rolled JSON via a `jsonEscape` helper rather than `JSONEncoder` for transparency at the boundary the model sees. **Timeout enforcement uses a shared-state actor pair** (`ExitGate`, `TimeoutFlag`) rather than racing two tasks' return values — the termination handler always fires when the process exits (including as a side-effect of `terminate()`), so racing return values picks the wrong winner ~half the time and reports `timed_out: false` for a process the timer just killed. **No sandbox beyond the user's macOS account.** `python.run` runs as the same user with full filesystem and network access; matches what `fs.write` already allows. Tool description warns explicitly. No `pip install` at runtime — packages are baked in at framework-build time; runtime install changes the security model and reproducibility story. **No `binaryTarget` for the framework** — SPM `binaryTarget` requires `.xcframework` / `.zip` / `.artifactbundle`, not a plain `.framework`. Args: `{path, pageRange?}` where `pageRange` accepts `"3"`, `"2-4"`, `"1,3-5,7"`, or `"all"` (default; pages are 1-indexed). Returns text with `--- Page N ---` separators between pages, capped at 256 KB with a truncation marker so a 500-page PDF can't blow the model's context in one call (model can re-call with a narrower range). Image-only / scanned PDFs (every selected page comes back empty) surface an explicit "OCR is required" error rather than empty success, so the model doesn't claim it read the document. Page-range parser is a separate static (`PageRangeOutcome` enum, fully unit-tested) — rejects `"0"` (1-indexed), reversed ranges (`"5-2"`), non-numeric garbage; out-of-range pages clamp silently. Uses `PDFKit` directly (Apple system framework, no fetch step, no licensing surface). 21 unit tests using real in-memory PDFs generated via `CGContext` + `CTLine` so the extraction path is exercised end-to-end.

- **`fs.write` builtin tool.** Sandboxed file write under the same allowed-roots model as `fs.read`. Args: `{path, content, overwrite?: bool=false}`. Atomic write via `Data.write(to:options:.atomic)` — content lands in a sibling temp file first and is renamed onto the target, so a partial write can't leave a half-written file under the user's eyes. Refuses to overwrite an existing file unless `overwrite: true` is passed (defensive default — accidental clobbers of a user's notes are exactly the failure mode worth one extra arg to avoid). Refuses to create parent directories: typo'd subdirs would silently create wrong-cased paths. Caps content at 1 MB. Returns `wrote N bytes to <path>` so the model can confirm the operation and cite the path. 9 unit tests.

- **`fs.list` builtin tool.** Sandboxed directory listing. Args: `{path, recursive?: bool=false, extensions?: [string], includeHidden?: bool=false}`. Returns a JSON array of `{path, name, isDirectory, size}` entries, alphabetically sorted (FS-order is unstable across runs and would make the tool's output non-deterministic). Bounded by 200 entries (truncation marker is a `{"truncated": true}` element appended to the array) and depth 4 in recursive mode. Hides dotfiles by default — `.git` / `.DS_Store` are noise for the typical "list the markdown notes" prompt. Extension filter accepts both `"md"` and `".md"` forms; applies to files only, so directories pass through during a recursive walk regardless of filter. 12 unit tests.

- **`math.compute` builtin tool.** Deterministic arithmetic via `NSExpression`. Args: `{expression}`. Models — especially small ones — silently miscompute multi-step arithmetic (`0.0825 * 12 * 30` is the canonical failure mode); one tool call eliminates the failure mode for any agent that opts in. Security: `NSExpression`'s `format:` parser exposes a `FUNCTION:` form that can invoke arbitrary Objective-C selectors at evaluation time. Mitigated by a strict whitelist regex (`^[0-9eE+\-*/().,\s]+$`) applied BEFORE `NSExpression` sees the input — letters can't appear, so `FUNCTION` / `SELF` / variable references are unreachable. Coerces bare integer literals to doubles before evaluation (`1 / 3` → `0.333…`, `1 / 0` → `+inf` which the `isFinite` guard rejects); without coercion `NSExpression` falls back to integer division (`1/3` → `0`) which is never what a model — or a user typing into a calculator — expects. Result formatted as integer when round (`100/4` → `"25"`, not `"25.0"`); float otherwise. 16 unit tests including the load-bearing `FUNCTION:` rejection and the integer-division non-truncation cases.

- **`clipboard.get` / `clipboard.set` builtin tools.** Read and replace the macOS clipboard via `NSPasteboard`. `get` ignores any args and returns the clipboard's `.string` representation (empty string when the pasteboard holds only image / file URL / non-text data — not an error, since the model's recovery is the same either way). `set` takes `{text}`, calls `clearContents()` before write so prior representations (RTF, file URL) don't survive alongside the new string and confuse paste targets, caps content at 64 KB (matches `fs.read`'s cap so the read-file → put-on-clipboard pattern is symmetric). Pasteboard is injectable via initialiser; chat VM wires `NSPasteboard.general`, tests use a private `NSPasteboard(name: UUID)` so `make test` doesn't trample the user's clipboard. 8 unit tests including the round-trip set→get.

  - Wired into `ChatViewModel.bootstrapAgents`: builtin tools register first, then `MCPHost.bootstrap` runs against the user's `mcp/` directory, then the merged `[ToolSpec]` catalog flows into `AgentController.bootstrap` exactly as before. `AppDelegate.applicationWillTerminate` calls `MCPHost.shutdown` so child processes don't outlive the app.
  - Tests: `MCPProtocolTests` (15 tests) covers JSON-RPC encoding (id/method/params, omitted-when-nil params, no-id notifications, response result/error decode), the v1 MCP shapes (tool list, content blocks, args parsing with bad-JSON fallback), `AnyJSON` round-trip + integral-double coercion, and `MCPServerConfig` decoding (minimal + full). `MCPClientTests` (7 tests) drives the full lifecycle through a `MockMCPTransport` (records sent frames, exposes `respond` for synthetic server replies): handshake sequence (initialize → notifications/initialized → tools/list), `callTool` text-block concatenation, `isError: true` mapping to `MCPError.toolErrored`, RPC-error mapping, pre-start guard, shutdown draining pending requests, and the `MCPBuiltinTool` adapter wrapping errors as `ToolResult(error:)`.
  - Out of scope for v1 (deferred): MCP resources / prompts / sampling capabilities, HTTP/WebSocket transports, in-app server config UI, hot-reload, MCP roots filesystem permissions abstraction.

- **`PlannerAgent` (item 10).** LLM-driven planner that owns its own loop via `customLoop`. Algorithm: (1) decode a planning prompt and parse the response as a numbered list into a `PlanLedger`; (2) execute each step by re-decoding with the rendered ledger as context — the model emits a tool call from the catalog or a brief text result; (3) on tool failure, replan once by asking the model to issue fresh steps that replace the remaining work (`PlanLedger.revise` overwrites everything from the cursor on, ordinals continue); (4) decode a final synthesis answer from the completed step outputs. Three knobs in `Config`: `maxStepDecodes` (hard cap on per-step decode rounds across one turn) and `maxReplans` (per-turn cap on plan revisions). Budget exhaustion produces a `.budgetExceeded` terminator and a deterministic synthesis from collected outputs so the user never sees a blank reply. The trace records every plan / execute / replan / synthesis event with distinct `assistantText` markers so the disclosure UI shows each phase. **Out of scope on purpose**: sub-agent dispatch (covered by composition primitives), branching plans (linear-with-replanning is the v1 shape), and a JSON-backed planner schema (Swift conformance keeps the policy readable). Closes the §5.2 / item 10 gap between curated workflows and a real agent runtime.

- **`PlanLedger` and `PlanParser` value types in `InferAgents`.** The planner's authoritative state for one turn: `goal`, ordered `[PlanStep]` with status (`pending` / `running` / `completed` / `failed` / `skipped`) + `output` + `errorMessage`, a `cursor` advancing through the steps, and a `replanCount` to cap thrash. Mutators (`beginCurrentStep`, `completeCurrentStep`, `failCurrentStep`, `revise(remainingSteps:)`) are non-throwing — invalid transitions are no-ops so a misbehaving model can't crash the planner. `Codable` so the ledger survives JSON round-trips and can be embedded into trace observability surfaces. `renderForPrompt()` produces a compact, glyph-prefixed (`[ ]`/`[~]`/`[x]`/`[!]`/`[-]`) status block embedded in per-step decode prompts so the model sees prior outputs as context. `PlanParser.parseSteps(from:)` is a tolerant parser for the loose formats LLMs actually emit — numbered (`1.` / `1)`), bulleted (`-` / `*` / `•`), or bare lines with preamble heuristics — returning an empty array when nothing recognisable is present so callers can fall back to a single-step plan.

- **`AgentDecoder` hook on `AgentContext`.** New `@Sendable (messages, params) async throws -> String` typealias and `decode: AgentDecoder?` property. Required by `customLoop` agents that need to talk to an LLM without a reference to the runner — the planner is the canonical caller. Returns the fully-accumulated assistant body (no streaming) because planner-style agents need whole structured output and adding streaming here would force every customLoop caller to consume an `AsyncThrowingStream` it doesn't want. `BasicLoop` populates the hook from its `runner` parameter before invoking `customLoop`; the chat-VM `runCustomLoopIfAvailable` populates it from the active `LlamaRunner` / `MLXRunner` actor (both already conform to `AgentRunner`). Deterministic / tool-only agents (`DeterministicPipelineAgent`) leave the hook unused. `AgentError.decoderMissing` is the hard-error case for an LLM-driven custom loop running without a wired decoder.

- **`StepTrace.TurnTelemetry`.** Per-turn measurements stamped by the loop driver: net tokens decoded, segment wall-clock duration, per-tool elapsed times (summed across calls to the same tool inside a turn), tool-call count, and tool-failure count. `Codable` and omitted from the encoded form when `nil` so legacy traces round-trip byte-identical. `Generation.runOneAgentTurn` populates the values on every return path (success, cancellation, error) by snapshotting `netTokenCount` and a `Date` at segment start; `maybeRunToolLoop` wraps each `toolRegistry.invoke` call with a wall-clock measurement that lands in the trace's `toolLatencyMillisByName` map. Surfaced inline in the chat transcript via a new `TelemetryBadge` chip next to the `StepTraceDisclosure` header — token count, duration, summed tool latency, and a red failure counter when non-zero, with a hover tooltip showing the per-tool breakdown. 5 new tests in `TurnTelemetryTests` covering count derivation, latency accumulation, JSON round-trip, and backward-compat decoding of legacy traces.

## [0.1.7]

### Added

- **Reasoning-model think-block handling.** Qwen3 / DeepSeek-R1 style `<think>…</think>` blocks are now rendered as a collapsed-by-default "Thoughts" disclosure above the assistant reply instead of leaking into the visible body as raw tag soup. `InferCore/ThinkBlockStreamFilter.swift` is a stateful stream filter that splits streaming pieces into `think` vs `reply` fragments and tolerates tags straddling piece boundaries (tag-start can arrive in one piece, tag-end in a later one). The transcript stores the two fragments side by side on `ChatMessage` and `ThinkingDisclosure` renders them with the same styling idiom as `StepTraceDisclosure`. Collapsed by default on every load — reasoning verbosity is interesting once, noise afterwards.

- **Thinking budget setting (`InferSettings.thinkingBudget`, default 4096).** Reasoning models decode both the `<think>…</think>` pass and the final answer; capping at `maxTokens` often exhausts the cap inside the think block and leaves nothing for the reply. The runner cap is now `maxTokens + thinkingBudget` — `maxTokens` stays the hard cap on *net* rendered tokens, `thinkingBudget` is the invisible decode allowance. Surfaced as a slider in the Parameters sidebar with guidance; persisted under `PersistKey.thinkingBudget`. For non-reasoning models the extra budget is harmless — they never emit think blocks so the net cap fires first.

- **Net-token accounting + KV compaction.** The generation loop now tracks net (rendered) tokens separately from total decoded tokens, and `requestStopCurrentRunner` fires when net reaches `maxTokens` even if the model is still inside a think block. After a turn that emitted thinking, `compactKVForVisibleHistory` clears the llama KV cache and re-submits the visible-only transcript, so think tokens don't eat context on subsequent turns. Header stats split when they diverge: `123 net · 456 gen · 45.2 tok/s`; identical counts collapse back to the compact single form.

- **Release-configuration Makefile targets.** `make build-release`, `make bundle-release`, `make run-release` recursively invoke the corresponding target with `INFER_CONFIG=Release`. Debug stays the default for active development (fast incremental builds, full debug symbols); Release is for perf testing, distribution dry-runs, or diagnosing "feels slower than it should" regressions against the optimized build. One set of rules, two configurations — no logic duplication.

- **Hybrid retrieval (vector + FTS5).** Dense embedding alone misses chunks whose answer vocabulary doesn't overlap with the query (e.g. a query for "vector database" against a corpus that uses the proper noun "SQLiteVec"). The vector store now keeps an FTS5 index over `chunks.content` (`content='chunks', content_rowid='id'` — same external-content pattern the vault uses on messages, with `unicode61 remove_diacritics 2` tokenizer). Triggers keep the index in sync on insert/update/delete; a one-shot backfill at bootstrap covers chunks ingested before hybrid existed. `VectorStore.search` runs both retrievers in parallel and fuses via Reciprocal Rank Fusion (k=60, equal weights) — chunks appearing in both retrievers' top lists rise above chunks appearing in only one. FTS-only hits get a real cosine distance via a `vec_distance_cosine(stored_embedding, query_embedding)` subquery so the UI similarity display stays consistent across retrievers. Per-search diagnostics on `VectorStore.lastSearchDiagnostics` (`vectorHits`, `ftsHits`, `ftsQuery`, `ftsError`, `usedFusion`); the Console tag `[hybrid: 30v+30f]` / `[vector-only: fts returned 0 for '...']` / `[fts error: ...]` makes it obvious which retriever contributed on each turn. Sanitizer is OR-joined and tokenizer-mirrored — splits the query on anything non-alphanumeric (matching `unicode61`'s tokenization of stored content), drops single-character tokens, phrase-quotes each term against FTS5 reserved-word collisions, joins with `OR` so BM25 can rank by partial matches rather than requiring every term in every chunk. 11 new unit tests cover the RRF fusion logic + sanitizer behaviour.

- **HyDE query reformulation (per-workspace, opt-in).** Before retrieval, the chat model generates a hypothetical passage answering the user's question, and *that* hypothetical is embedded for dense retrieval (FTS still uses the original query — keyword search benefits from the user's actual terms). Bridges vocabulary gaps when query phrasing diverges from source phrasing. New `generateOneShot(prompt:maxTokens:)` method on both runners isolates the side-channel call from the main conversation: `LlamaRunner` lazy-initializes a secondary llama context (separate KV cache, low-temperature sampler) so the user's chat state isn't disturbed; `MLXRunner` instantiates a fresh `ChatSession` with empty history. Per-workspace toggle stored in `UserDefaults` under `infer.workspace.<id>.hydeEnabled`; off by default since HyDE's quality ceiling is the chat model's knowledge of the corpus domain — for specialized corpora the chat model only knows at summary level (e.g. a long novel + small quantized chat model), the hypothetical can hallucinate and *misdirect* retrieval. Console tag `[hyde]` confirms when the path fired; the hypothetical itself is logged at debug level (source `rag`) so users can inspect what the model produced.

- **Cross-encoder reranking (per-workspace, opt-in).** When enabled on a workspace, hybrid retrieval over-fetches to 30 candidates and a `bge-reranker-v2-m3` cross-encoder re-scores each (query, chunk) pair, returning the top 5 by reranker score. Catches "topical-but-not-answerful" chunks the dense+FTS fusion couldn't separate. New `RerankerRunner` actor parallels `EmbeddingRunner` — same llama.cpp context pattern but configured with `LLAMA_POOLING_TYPE_RANK`, reads concatenated `[BOS] query [SEP] doc [SEP]` token streams, gets a single relevance logit per pair via `llama_get_embeddings_seq(ctx, 0)[0]`. Sequential under the hood (~1–2s for 30 pairs on M-series); batching deferred. New `RerankerModelRef` enum pins `gpustack/bge-reranker-v2-m3-GGUF` / `bge-reranker-v2-m3-Q8_0.gguf` (~315 MB), downloaded on demand through the same Hugging Face flow as the embedder with a missing-model banner in the workspace sheet. Console tag `[rerank: top5 of 30 in Ts, best=X.XX]` reports time + the reranker's top score (raw logit; positive = relevant, negative = not — useful diagnostic when the reranker's "best" is highly negative, indicating none of the candidates strongly answer the query). Toggle is disabled in the UI until the model is downloaded; the download banner shows even with the toggle off so users have a discoverable path to enable the feature.

- **Per-workspace settings infrastructure** (`PersistKey.workspaceKey(id:setting:)` + `PersistKey.WorkspaceSetting`). UserDefaults-backed key shape `infer.workspace.<id>.<setting>` so new per-workspace toggles can be added without a vault schema change. Two settings ship: `hydeEnabled`, `rerankEnabled`. Surfaced in the workspace sheet's new "Retrieval quality" section with one-line descriptions of each toggle's latency trade-off.

- **Console-side observability for the RAG pipeline.** Every retrieval-related path logs structured events under sources `rag`, `embedding`, `rerank`, `workspaces`. Per-file scan progress at debug level (`ingested X (N chunks)`) so users at the debug filter can verify nothing silently dropped; explicit terminal `scan successful` vs. `scan completed with warnings` line so success isn't ambiguous. Empty-chunks skip (rare — file contains only separator characters) now logs a warning instead of being a silent skip. `RAG.initialize()` happens in `applicationDidFinishLaunching` and the FTS index health check (`chunks=N, chunks_fts=M`) runs at VM init so a backfill mismatch is visible immediately.

- **Context-window percentage in the chat header (llama only).** Compact `42%` indicator next to the per-generation tok/s readout; tints orange past 80%, red past 95%. Tooltip carries raw `Context: used / total tokens used`. Renders only when the backend exposes a real context size — llama via `llama_n_ctx`; MLX doesn't expose `max_position_embeddings` from the loaded `ModelContainer`, so the indicator stays absent there (consistent with the rest of the MLX header). - **Retrieval-augmented generation (RAG), per-workspace.** A workspace's data folder becomes its corpus: files in `.txt` / `.md` / `.json` get ingested into a dedicated vector store, and every chat in that workspace augments user turns with retrieved context before generation. End-to-end flow in six composable pieces:
  - **Vector store** (`InferRAG` target, new library): actor over [`SQLiteVec`](https://github.com/jkrukowski/SQLiteVec) backed by a separate `vectors.sqlite` file. `ensureInitialized` stamps per-workspace metadata (embedding model, dimension, metric, chunk size/overlap) and hard-fails on mismatch to prevent silent index corruption. `ingest` is transactional and idempotent on `(workspace_id, content_hash)`. `search` runs KNN via the `vec0` virtual table scoped to the workspace. `deleteWorkspaceData` cascades manually across `sources` → `chunks` → `vec_items` since sqlite-vec virtual tables don't participate in FK chains. Dimension is baked into the `vec0 float[384] distance=cosine` declaration to match bge-small.
  - **Embedding runner** (`Infer/EmbeddingRunner.swift`): dedicated llama.cpp context configured for embedding (`LLAMA_POOLING_TYPE_NONE`, `embeddings = true`, encoder mode), running a GGUF embedding model independently of the chat `LlamaRunner`/`MLXRunner` so the model stays resident across queries. Mean-pool + L2-normalize on the Swift side via `Accelerate.vDSP` — per the cyllama port's experience, llama.cpp's internal pooling is unreliable across model families. Non-blocking busy flag rejects concurrent re-entry. `llama_memory_clear` between calls so sequential embeds don't leak KV state.
  - **Embedding model discovery + Hugging Face download** (`Infer/EmbeddingModel.swift`): single `EmbeddingModelRef` enum pins the default (`CompendiumLabs/bge-small-en-v1.5-gguf`, `bge-small-en-v1.5-q8_0.gguf`, 384 dim, ~130 MB). Model lands in the same GGUF directory as chat models. Missing-model banner in the workspace sheet offers a one-click download via `HubClient.downloadFile` with progress routed through `Progress` KVO → `vm.embeddingModelDownloadProgress` → an inline progress bar. No launch-time nag — the download only appears once a user engages with a RAG-enabled workflow.
  - **Text splitter** (`InferRAG/TextSplitter.swift`): recursive character splitter ported from cyllama. Hierarchical separators (`\n\n` → `\n` → `. ` → ` ` → hard split), configurable size/overlap (defaults 512/50), grapheme-accurate offsets preserved for citation UI, merge phase seeds each new chunk with the tail of the previous for character-level overlap. 12 unit tests cover paragraphs, sentences, hard-split fallback, overlap invariant, Unicode/emoji, monotonic offsets, size ceilings.
  - **Ingestion orchestrator** (`Infer/ChatViewModel/Ingest.swift`): `scanAndIngest(workspaceId:)` enumerates the data folder recursively (skipping hidden files and package contents), hashes each file via MD5 for dedup, loads (`SourceLoader` in `RAGLoaders.swift` — UTF-8 `.txt`/`.md`/`.json`, rejects empty files), splits into chunks, embeds each chunk sequentially, and stores transactionally. Live progress published on `vm.ingestProgress` with `(totalFiles, processedFiles, currentFile, ingested, skippedDuplicates, failed)`; workspace sheet renders a progress bar + current file + per-counter split while running, and a corpus stats line (`N sources, M chunks`) when idle. Per-file failures are warnings in `LogCenter`; only catastrophic failures (embedder load, vector store init) abort the scan.
  - **Query pipeline** (`Infer/ChatViewModel/RAGPipeline.swift`): `runRAGIfAvailable(userText:)` runs on every user turn when the active workspace has an indexed corpus. Embeds the query, top-5 KNN, filters by cosine distance ≤ 1.2, prepends a context block to the user message (not the system prompt — agent personas keep full control of their system prompt). The original user text is what's persisted to the vault; the augmented prompt is what hits the runner. Every failure path downgrades silently to "no augmentation" so retrieval failures never block replies.
  - **Citations** (`Infer/ChatView/ChatTranscript.swift`): assistant messages produced via RAG render a collapsed "Sources" disclosure mirroring `StepTraceDisclosure`'s styling. Header shows source count and best-match similarity (`1 - distance/2` under cosine). Expanded rows show filename, chunk ordinal, distance, preview text (selectable), and a Reveal-in-Finder button. `ChatMessage.retrievedChunks` is ephemeral — session-only, not persisted in the vault (history view shows messages without retrieval provenance in MVP).

- SwiftUI chat app with two inference backends selectable at runtime via a header picker.

  - `llama.cpp` backend: loads a local `.gguf` file through `llama.xcframework` (downloaded by `scripts/fetch_llama_framework.sh`).

  - MLX backend: loads any MLX-compatible Hugging Face repo id via `mlx-swift-lm` (Mode 1 integration using `#huggingFaceLoadModelContainer`); defaults to `LLMRegistry.gemma3_1B_qat_4bit`.

- `scripts/fetch_llama_framework.sh`: downloads a tagged `llama-<tag>-xcframework.zip` from the llama.cpp releases and installs `thirdparty/llama.xcframework` (default tag `b8848`).

- Makefile targets: `fetch-llama`, `build-infer`, `bundle-infer`, `run-infer`.

- Markdown rendering for assistant messages via `swift-markdown-ui` with the `gitHub` theme.

- Swift syntax highlighting for fenced code blocks via `Splash` (other languages render monospaced/themed but uncolored).

- Hyperlinks in assistant messages open in the default browser (custom `OpenURLAction` routing to `NSWorkspace.shared.open`).

- Auto-load the last successfully loaded model on app launch (backend + path/HF id persisted in `UserDefaults`).

- `Edit > Copy Transcript as Markdown` (⇧⌘C): copies the full session as markdown (one `## <role>` block per message, separated by `---`).

- `File > Print Transcript…` (⌘P): renders the transcript through `swift-markdown`'s `HTMLFormatter` into a styled HTML document, loads it in an off-screen `WKWebView`, snapshots via `createPDF(configuration:)`, and prints through `PDFDocument.printOperation` for reliable pagination.

- Header settings popover (gear icon) with:

  - System prompt (multi-line `TextEditor`), applied to `ChatSession(instructions:)` on MLX and prepended as role=`system` on llama.

  - `temperature` and `top_p` sliders — rebuild the llama sampler chain / MLX `GenerateParameters` without requiring re-load.

  - `max_tokens` stepper, applied per-send on both backends.

  - Settings persist across launches in `UserDefaults`.

- Determinate progress bar for MLX model downloads via the `progressHandler` variant of `#huggingFaceLoadModelContainer`.

- `Cancel` button in the header while a load is in flight; cancels the load `Task` and rolls the UI back to the pre-load state.

### Changed

- Migrated from CMake to Swift Package Manager.

- `make build-infer` uses `xcodebuild` (not `swift build`) because mlx-swift's Metal kernels can only be compiled by Xcode's Metal toolchain; this requires the separate Metal Toolchain component (`xcodebuild -downloadComponent MetalToolchain`, ~700 MB, one-time).

- `bundle-infer` embeds `llama.framework` (from the `macos-arm64_x86_64` xcframework slice) into `Infer.app/Contents/Frameworks/` and copies SPM-emitted resource bundles (including `mlx-swift_Cmlx.bundle` containing `default.metallib`) into `Infer.app/Contents/Resources/`.

- Transcript background now uses `Color(.textBackgroundColor)` so it follows the system light/dark appearance.

### Notes

- First MLX model load downloads to `~/.cache/huggingface/hub/` (standard HF cache layout; override with `HF_HOME`).

- Prerequisites: Xcode 16.3+ / Swift 6.1+ (required by `mlx-swift-lm`).

- Load cancellation for the llama backend is cooperative: `llama_model_load_from_file` is a synchronous C call, so `Task.cancel()` only takes effect at the next Swift await point (fine for HF downloads, best-effort for large `.gguf` mmap).

- Changing the system prompt resets the conversation on both backends (history is lost); sampler param changes (`temperature`, `top_p`) do not.
